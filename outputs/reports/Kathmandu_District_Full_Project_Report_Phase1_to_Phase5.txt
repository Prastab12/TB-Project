ABSTRACT
================================================================================

This report presents a complete, end-to-end account of the Nepal NTP TB FHIR
Integration Project for Kathmandu District — from raw DHIS2 data ingestion and
quality assessment through FHIR R4 resource generation, REST API deployment,
public health dashboard development, and the formal publication of a FHIR R4
Implementation Guide (IG) as the capstone Phase 5 deliverable.

Over five Bikram Sambat (BS) fiscal years (2078–2082), Kathmandu District
contributed 60 monthly TB reporting records covering 13 performance indicators.
These records underwent rigorous quality assessment, standardised cleaning,
epidemiological validation, and complete transformation into 780 FHIR R4
MeasureReport JSON resources — each annotated with a custom NepaliCalendarPeriod
extension that encodes the BS calendar context alongside mandatory Gregorian dates.

The project resolves a fundamental interoperability gap: Nepal's National
Tuberculosis Programme (NTP) captures aggregate TB data in DHIS2 using the
Bikram Sambat calendar, while the international health informatics standard
(HL7 FHIR R4) natively supports only Gregorian ISO 8601 dates. The solution
— a custom FHIR extension with a Python ETL pipeline, FSH-defined profiles,
and a published Implementation Guide — creates a standards-compliant, machine-
readable, and internationally interoperable TB performance record for Kathmandu.

Kathmandu's dataset is one of the cleanest in the Bagmati Province cohort:
zero over-reporting errors, zero negative values, complete treatment outcome
and ART coverage records for all five years, and missingness confined entirely
to a known three-month structural reporting gap in BS 2078. The FHIR IG
Publisher validated 709 HTML artifact pages with zero structural profile errors.
All 77,368 internal validation checks passed with 100% accuracy.

  FHIR Data Absent Reason (DAR) encoding is used in Phase 4 instead.

3. PHASE 2 — DATA CLEANING AND STANDARDISATION
================================================================================

3.1 Cleaning Strategies Applied
---------------------------------
Three cleaning strategies were applied across the pipeline. Their application
to Kathmandu's 60 rows is documented below.

STRATEGY A — Zero-Filling Missing Values:
  Scope applied to Kathmandu:
    36 cells across 3 under-reported months (BS 2078 Baishak, Jestha, Asar)
    1 additional cell (relapse_total, BS 2079 Magh)
  Total zero-fills for Kathmandu : 37 cells
  Rationale: Prevents NaN propagation in downstream FHIR pipelines where
  null integer fields would cause serialisation failures.
  Critical note: These zeroes are NOT interpreted as true-zero TB reports.
  The UNDER_REPORTED set tracks them programmatically. Phase 4 replaces
  the zero values with FHIR Data Absent Reason annotations, ensuring that
  no misleading epidemiological signal enters the FHIR resources.

STRATEGY B — Int64 Type Enforcement:
  All count columns converted from float64 to pandas Int64 (nullable integer).
  Columns affected: new_cases_total, new_cases_female, new_cases_male,
    relapse_total, relapse_female, relapse_male, total_tb_m+f, pbc_reg,
    cured, failed, died, ltfu, not_eval
  All 60 Kathmandu rows passed type conversion without exception.
  Result: Zero float representations of whole numbers in final data.

STRATEGY C — District Name Standardisation:
  Kathmandu required no correction. The name "KATHMANDU" was already
  consistent and correctly cased across all 60 rows in the source file.
  (Corrections were applied to MAKWANPUR and CHITWAN for other districts.)

4. PHASE 3 — HL7 FHIR ARCHITECTURAL MAPPING
================================================================================

4.1 FHIR Module Selection
---------------------------
The HL7 FHIR Clinical Reasoning Module was selected as the interoperability
framework for this project. Within this module, two resource types form the
core mapping architecture:

  Measure (Definitional resource):
    A computable quality standard that defines the indicator, its scoring
    type, population criteria, and canonical URL. Measure resources are
    district-agnostic — they define WHAT is measured, not where or when.
    One Measure definition exists per TB indicator (13 total).

  MeasureReport (Instance resource):
    Holds the calculated result for a specific indicator, in a specific
    location, for a specific time period. References its parent Measure by
    canonical URL. Each MeasureReport instance represents one month of
    data for one indicator in Kathmandu District.
    Total: 780 instances (13 indicators × 60 months).

4.2 Scoring Type Architecture
-------------------------------
Two FHIR scoring types are applied based on the mathematical nature of
each indicator:

RATIO Scoring — 7 Monthly Notification Indicators:
  FHIR scoring code : "ratio"
  Populations       : 2 per group (numerator + denominator)
  Numerator         : Monthly TB case count from NTP reporting
  Denominator       : CBS mid-year district population (not a TB cohort)
  MeasureScore      : numerator / denominator (dimensionless, unit = "1")
                      Multiply by 100,000 to obtain rate per 100,000 population
  Frequency         : Monthly (60 distinct values per indicator)

COHORT Scoring — 6 Annual Treatment Outcome Indicators:
  FHIR scoring code : "cohort"
  Populations       : 1 per group (initial-population only)
  Count             : Annual treatment cohort count from NTP records
  MeasureScore      : Not calculated (counts only; rates computed in dashboard)
  Frequency         : Annual value repeated across all 12 months of fiscal year

4.3 Indicator Mapping Table
-----------------------------

  Measure ID                    | Indicator Title            | CSV Numerator Column
  ──────────────────────────────|────────────────────────────|─────────────────────

4.5 Reference Architecture
----------------------------
All MeasureReports use EXTERNAL references (not contained resources).

  subject  → Reference("Location/loc-kathmandu")
  reporter → Reference("Organization/org-ntp")
  measure  → Canonical URL string

Rationale: External references give each shared entity a stable, independently
resolvable canonical URL. Contained resources cannot be independently addressed
by FHIR servers and are inappropriate for shared institutional entities.

5.3 NepaliCalendarPeriod Extension
------------------------------------
A custom FHIR R4 extension was designed to carry Nepal's Bikram Sambat calendar
metadata alongside mandatory Gregorian ISO 8601 dates in every MeasureReport.

  Extension URL  : https://iihms.gov.np/fhir/StructureDefinition/nepali-fiscal-period
  Context        : Applied to MeasureReport.period (FHIR Period type)
  Applied to     : All 780 MeasureReport resources

  Sub-extensions (all 1..1):
    bs-year      (valueInteger) : BS year, e.g. 2078
    bs-month     (valueString)  : BS month name, e.g. "Shrawan"
    fiscal-year  (valueString)  : Nepali fiscal year, e.g. "2078/79"

  Gregorian date convention:
    period.start = {gregorian_year}-{gregorian_month}-16
    period.end   = {gregorian_year'}-{gregorian_month+1}-15

  BS-to-Gregorian month mapping:
    Baishak→4, Jestha→5, Asar→6,   Shrawan→7, Bhadra→8,  Ashwin→9
    Kartik→10, Mangsir→11, Poush→12, Magh→1,   Falgun→2,  Chaitra→3

  Gregorian year offset:
    Months 4–12 (Baishak–Chaitra): g_year = bs_year - 57
    Months 1–3  (Magh–Asar):       g_year = bs_year - 56

  Fiscal year logic:
    Months Shrawan–Chaitra (FY positions 1–9): FY = {year}/{year+1 mod 100}
    Months Baishak–Asar    (FY positions 10–12): FY = {year-1}/{year mod 100}
    Example: BS 2078 Shrawan → "2078/79", BS 2078 Baishak → "2077/78"

5.4 Data Absent Reason (DAR) Encoding
---------------------------------------
Under-reported months and isolated empty cells are annotated using the
FHIR standard Data Absent Reason extension. This is the correct FHIR R4
mechanism for encoding structurally expected but temporarily unavailable
data on primitive fields.

  Extension URL : http://hl7.org/fhir/StructureDefinition/data-absent-reason
  valueCode     : "not-reported"

  Applied to numerator population count (integer primitive extension):
    "_count": {
      "extension": [{
        "url": "http://hl7.org/fhir/StructureDefinition/data-absent-reason",
        "valueCode": "not-reported"
      }]
    }

  Applied to measureScore value (decimal primitive extension):
    "measureScore": {
      "unit": "1", "system": "http://unitsofmeasure.org", "code": "1",
      "_value": {
        "extension": [{
          "url": "http://hl7.org/fhir/StructureDefinition/data-absent-reason",
          "valueCode": "not-reported"
        }]
      }
    }

  Note: The denominator (CBS population) is ALWAYS present even in DAR months
  because CBS population estimates are external to DHIS2 and always available.
  Only the numerator (TB case count) and derived measureScore are DAR-flagged.

  DAR application summary:
    3 under-reported months × 7 ratio indicators  : 21 MeasureReports
    1 isolated empty cell (relapse_total 2079 Magh): 1 MeasureReport
    Total DAR-annotated MeasureReports             : 22
    Cohort indicators with DAR                     : 0 (all complete)

6.9 Python ETL Pipeline (IG Mode)
-----------------------------------
File: ig/scripts/etl.py
Dependencies: fhir.resources==8.2.0, pandas>=2.0.0

The ETL pipeline reads final_cleaned_data.csv (60 rows) and generates
type-safe FHIR R4 MeasureReport JSON using the fhir.resources library.

Data flow:
  final_cleaned_data.csv
       ↓ pandas.read_csv() → DataFrame (60 rows)
       ↓ row iteration (13 indicators × 60 months)
       ↓ fhir.resources model construction
       ↓ _to_r4() post-processing (R5 key rename)
       ↓ json.dumps(indent=2)
       → ig/fhir-output/measure-reports/by_indicator/  (780 files, gitignored)
       → ig/input/resources/                           (3 IG example files)

Key functions:
  _iso_period(bs_year, bs_month) → (start_date, end_date):
    Converts BS year+month to Gregorian ISO date pair using month-to-GM map.
    Handles year-wrapping for Magh, Falgun, Chaitra (Gregorian months 1–3).

  _fiscal_year(bs_year, bs_month) → "YYYY/YY" string:
    Uses ordered fiscal year list to determine the correct FY string.

  _ratio_group(row, ind_id, num_col, bs_year, bs_month) → MeasureReportGroup:
    For normal months: builds numerator (count=N) + denominator (count=D)
      populations and measureScore Quantity(value=N/D, unit="1").
    For DAR months: builds numerator with _count DAR primitive and
      measureScore with _value DAR primitive.

  _cohort_group(row, ind_id, csv_col) → MeasureReportGroup:
    Builds single initial-population with count from CSV column.
    No measureScore constructed.

  _to_r4(obj) — R4 serialisation fix:
    fhir.resources 8.x uses the R5-style field name "measureScoreQuantity"
    internally. FHIR R4 wire format requires "measureScore".
    This recursive dict rename corrects the field name before writing JSON.
    _R5_TO_R4 = {"measureScoreQuantity": "measureScore"}

CLI modes:
  python3 etl.py             → Generate all 780 MeasureReports + 3 examples
  python3 etl.py --examples  → Generate only 3 IG examples (CI/CD fast mode)

8. DESIGN DECISIONS AND JUSTIFICATIONS
================================================================================

Decision 1 — External References (No Contained Resources)
  All 780 MeasureReports reference Organization/org-ntp and
  Location/loc-kathmandu as external references, never as contained resources.
  Justification: Contained resources lack a stable resolvable URL. External
  references allow each shared entity to have a canonical identity accessible
  independently by FHIR servers. FHIR best practice explicitly discourages
  contained resources for shared institutional data.

Decision 2 — Ratio vs Cohort Scoring Split
  Notification indicators use "ratio" scoring (numerator/denominator structure).
  Treatment outcome indicators use "cohort" scoring (initial-population only).
  Justification: This mirrors the mathematical distinction between incidence
  rate measures (which require a population denominator to compute a rate) and
  absolute outcome counts (where the cohort size IS the initial population).
  Machine-readable scoring types make the indicator semantics unambiguous.

Decision 3 — UCUM Unit "1" for MeasureScore
  All ratio measureScore values use UCUM code "1" (dimensionless).
  The stored value is numerator/denominator (e.g., 0.00018979).
  Justification: Consumers multiply by 100,000 to get CNR per 100k. Storing
  the raw ratio is more mathematically precise than storing per-100k values,
  and allows any scaling factor to be applied by the consuming system.

Decision 4 — DAR on Primitive Fields, Not Element Removal
  Data Absent Reason is applied to _count and _value (primitive extensions)
  rather than removing the population or score elements entirely.
  Justification: FHIR R4 §6.7 defines primitive extension (underscore prefix)
  for extending primitive types. Removing the count element entirely would
  break structural profile validation. The _count DAR pattern correctly
  communicates that the slot exists but the value is absent for a known reason.

Decision 5 — Plain-Text Population Criteria Language
  Measure.group.population.criteria.language is fixed to #text/plain.
  Justification: The source system (DHIS2) does not support CQL or FHIRPath.
  Population criteria describe DHIS2 data element names in plain language.
  Claiming CQL or FHIRPath compliance would be technically dishonest and
  would fail compliance validation against CQL parsers.

Decision 6 — Gregorian Date Encoding Convention
  period.start = {g_year}-{g_month}-16, period.end = {g_year'}-{g_month+1}-15
  Justification: BS months typically start around the 15th–17th of the
  Gregorian month. Using the 16th as a universal start date provides a
  consistent 30-day window aligned with NTP monthly reporting cycles.
  The NepaliCalendarPeriod extension preserves the exact BS context.

Decision 7 — fhir.resources 8.2.0 with _to_r4() Post-Processing
  The library uses R5-style field name "measureScoreQuantity" internally.
  Post-processing with _to_r4() renames it to "measureScore" (R4 name).
  Justification: fhir.resources 8.x is the only Python FHIR library fully
  compatible with Pydantic v2. Pinning to older versions would cause
  dependency conflicts. The post-processing rename is a clean, maintainable
  workaround with no impact on JSON structural validity.

Decision 8 — Annual Cohort Values Repeated Monthly
  Treatment outcome counts (pbc_reg, cured, etc.) are annual figures.
  The same annual value appears in all 12 monthly MeasureReports for the year.
  Justification: The DHIS2 source exports treatment outcomes as annual cohort
  totals, not monthly breakdowns. Monthly granularity is not available. Repeating
  the annual value 12 times preserves the monthly timeline structure required by
  MeasureReport while faithfully representing the source data resolution.

Decision 9 — Jekyll via ruby/setup-ruby in CI
  GitHub Actions uses ruby/setup-ruby@v1 (Ruby 3.3) to install Jekyll.
  Justification: Ubuntu's system Ruby is version 2.x — too old for Jekyll 4.x.
  The ruby/setup-ruby action is the official, maintained approach to installing
  a modern Ruby on GitHub Actions runners.

9. CHALLENGES AND SOLUTIONS
================================================================================

Challenge 1 — Space Character in macOS Directory Path
  Error   : "There is a space in the folder path: /Users/pathibhara/Desktop/
             TB Project/ig/"
  Cause   : IG Publisher v2.2.8 rejects spaces in any directory in the full path.
  Solution: Copied ig/ to /private/tmp/tb-ig2/ (canonical path, no symlinks,
            no spaces) and ran the publisher from there. The GitHub Actions
            runner's path (/home/runner/work/TB-Project/) has no spaces.

Challenge 2 — Duplicate hl7.fhir.r4.core Dependency
  Error   : "Name 'hl7.fhir.r4.core' already exists (value = '4.0.1')"
  Cause   : sushi-config.yaml had an explicit "hl7.fhir.r4.core: 4.0.1"
            dependency that the IG Publisher also loads automatically.
  Solution: Removed the explicit dependency. The publisher loads the core
            R4 package automatically without needing it declared.

Challenge 3 — fhir.resources 6.5.0 Pydantic v2 Incompatibility
  Error   : "ImportError: cannot import name 'ROOT_VALIDATOR_CONFIG_KEY'"
  Cause   : fhir.resources 6.x used Pydantic v1 APIs removed in v2.x.
  Solution: Upgraded to fhir.resources 8.2.0 (fully Pydantic v2 compatible).

Challenge 4 — R5 Field Name in fhir.resources 8.x
  Issue   : Library uses "measureScoreQuantity" (R5 polymorphic type name).
            FHIR R4 wire format requires "measureScore".
  Solution: Accept the R5 name in the model constructor, then apply
            _to_r4() recursive dict rename after model_dump_json() serialisation.
            _R5_TO_R4 = {"measureScoreQuantity": "measureScore"}

Challenge 5 — Jekyll Not Found by IG Publisher (Local)
  Error   : "Cannot run program 'jekyll': No such file or directory"
  Cause   : fhir.base.template requires Jekyll for markdown→HTML rendering.
            macOS system Ruby 2.6 is too old for Jekyll 4.x. System Ruby gems
            directory is read-only (/Library/Ruby/Gems/2.6.0).
  Solution: brew install ruby → installs Ruby 4.0.5 to /opt/homebrew/opt/ruby.
            gem install jekyll bundler via Homebrew Ruby.
            Export PATH="/opt/homebrew/opt/ruby/bin:/opt/homebrew/lib/ruby/gems/
            4.0.0/bin:$PATH" before running the IG Publisher.

Challenge 6 — Jekyll Not Found by IG Publisher (GitHub Actions)
  Error   : "Cannot run program 'jekyll': error=2, No such file or directory"
  Cause   : Ubuntu-latest runner lacks Jekyll; system Ruby too old.
  Solution: Added ruby/setup-ruby@v1 (Ruby 3.3) + gem install jekyll bundler
            steps to the workflow before the IG Publisher step.

Challenge 7 — GitHub Pages Deployment 404
  Error   : "Failed to create deployment (status: 404)"
  Cause   : GitHub Pages not enabled in repository settings.
  Solution: Repository Settings → Pages → Source → "GitHub Actions".
            One-time manual configuration, not a code fix.

Challenge 8 — DAR on Integer/Decimal Primitive Fields
  Issue   : FHIR R4 DAR must be applied as a primitive extension on _count
            and _value, not as a replacement of the element.
            fhir.resources model_validate() must be used for these fields.
  Solution: Used MeasureReportGroupPopulation.model_validate({"code": ...,
            "_count": dar_primitive()}) and Quantity.model_validate(
            {"_value": dar_primitive(), "unit": "1", ...}).

11. CONCLUSION
================================================================================

This report documents the complete five-phase Nepal NTP TB FHIR Integration
Project for Kathmandu District — from raw DHIS2 data extraction through the
publication of a formal, standards-compliant FHIR R4 Implementation Guide.

The project addressed a real-world interoperability challenge: Nepal's NTP
TB programme captures aggregate monthly data using the Bikram Sambat calendar
in DHIS2, while HL7 FHIR R4 — the international standard for health data
exchange — natively supports only Gregorian ISO 8601 dates and has no built-in
model for the BS calendar or Nepali fiscal year. No existing solution addressed
this gap.

The five-phase solution delivers:

Phase 1: Rigorous quality assessment confirming Kathmandu as the cleanest
dataset in Bagmati Province — zero over-reporting errors, zero negative values,
and complete coverage for all critical variables. The three-month structural
reporting gap in BS 2078 was identified and classified correctly as a DHIS2
system limitation, not a Kathmandu data quality failure.

Phase 2: Standards-compliant data cleaning using zero-fill with programmatic
tracking of under-reported months, Int64 type enforcement, and Pandera schema
validation — producing a 60-row, 18-column dataset verified ready for FHIR
transformation.

Phase 3: A well-reasoned FHIR Clinical Reasoning Module architecture using
Measure (definitions) and MeasureReport (instances), with ratio scoring for
7 notification indicators and cohort scoring for 6 treatment outcome indicators,
all referenced through a consistent canonical URL scheme under iihms.gov.np.

Phase 4: Complete generation of 780 type-safe FHIR R4 MeasureReport resources,
each carrying a custom NepaliCalendarPeriod extension preserving BS year, month,
and fiscal year context. Under-reported months are encoded using the FHIR Data
Absent Reason standard. All 77,368 cross-validation checks passed with 100%
accuracy. A FastAPI FHIR REST API and Chart.js dashboard bring the data to life.

Phase 5: A fully published FHIR R4 Implementation Guide with FSH-defined profiles
(TBMeasure, TBMeasureReport), a formally specified extension (NepaliCalendarPeriod),
a BS month CodeSystem and ValueSet, shared canonical instances (org-mohp, org-ntp,
loc-kathmandu), a Python ETL pipeline, GitHub Actions CI/CD automation, and 709
IG Publisher-generated HTML pages validated with zero structural profile errors.

Epidemiologically, Kathmandu District shows a positive five-year trajectory:
TB notification peaked in BS 2080–2081, treatment cure rates reached 87.1%
(exceeding the WHO 85% target), LTFU rates declined from 2.6% to 1.4%, and
the not-evaluated rate fell dramatically from 7.7% to 0.8% — all consistent
with a maturing, well-managed TB programme in Nepal's capital district.

The 780 FHIR R4 MeasureReport resources and their accompanying Implementation
Guide represent a complete, internationally interoperable, and semantically
annotated TB performance record for Kathmandu — capable of integration with
national health information exchange systems, international TB surveillance
platforms (WHO GDO, Global Fund reporting), and any FHIR R4-compliant system
worldwide.

The project demonstrates that Nepal's TB programme data — historically locked
in DHIS2 CSV exports using a non-Gregorian calendar — can be fully and faithfully
represented in HL7 FHIR R4 through thoughtful extension engineering, principled
ETL design, and a formally published Implementation Guide that other districts,
other disease programmes, and other countries using non-Gregorian calendars
can adopt as a reusable template.
