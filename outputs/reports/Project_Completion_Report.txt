ABSTRACT
================================================================================

This project systematically transforms five years of raw tuberculosis (TB)
surveillance data from Kathmandu District, Nepal — sourced from the Integrated
Health Information Management System (IIHMS) — into a standards-compliant HL7
FHIR R4 interoperable data layer. The work is structured across four completed
phases: (1) Data Ingestion and Quality Assessment, (2) Data Cleaning and
Standardisation, (3) HL7 FHIR Architectural Mapping, and (4) FHIR Transformation
and Extension Engineering.

The project covers 60 months of monthly TB programme reporting spanning BS 2078
Shrawan through BS 2082 Asar, across 13 districts of Bagmati Province, with
Kathmandu District as the primary analytical and interoperability focus. The
output comprises 780 FHIR R4 MeasureReport resources, 13 Measure definitions,
a custom FHIR extension for Bikram Sambat calendar preservation, a FastAPI-based
FHIR REST server, and an interactive public health dashboard — all conformant
to HL7 FHIR R4 specification (version 4.0.1).

1. PROJECT BACKGROUND AND OBJECTIVES
================================================================================

Tuberculosis (TB) remains a significant public health challenge in Nepal, with
the country classified among the 30 high-burden TB nations globally. The
National Tuberculosis Programme (NTP) under the Ministry of Health and Population
(MoHP) collects monthly aggregate TB surveillance data through DHIS2, stored
within IIHMS. However, this data exists in proprietary flat-file formats that
lack interoperability with global health information exchange standards.

The primary objective of this project is to bridge the gap between Nepal's
existing DHIS2-based TB surveillance infrastructure and the HL7 FHIR R4
interoperability standard, enabling machine-readable, semantically annotated,
and internationally exchangeable TB performance reporting.

Core objectives:
  - Ingest, profile, and assess the quality of five years of IIHMS TB data
  - Apply a reproducible, auditable data cleaning pipeline
  - Design a formal DHIS2-to-FHIR mapping architecture for TB indicators
  - Generate 780 FHIR R4 MeasureReport resources for Kathmandu District
  - Preserve Bikram Sambat (BS) calendar context through a custom FHIR extension
  - Expose the FHIR resources via a standards-conformant REST API
  - Visualise derived public health rates through an interactive dashboard

4.2 Cleaning Strategies Applied
---------------------------------

  STRATEGY A — Zero-Filling Missing Values:
    All empty cells in numeric columns were replaced with integer 0.
    Rationale: Prevents downstream NaN propagation errors in FHIR
    calculation pipelines and aggregate reporting. Cells that are
    genuinely under-reported (BS 2078 Baishak, Jestha, Asar for
    Kathmandu) are preserved as 0 and subsequently handled through
    the FHIR Data Absent Reason mechanism in Phase 4.
    Values filled: 762 cells across the 780-row dataset.

  STRATEGY B — Strict Int64 Type Enforcement:
    All numeric columns (case counts, population denominators,
    treatment outcomes) converted to Int64 type.
    26 columns type-converted.
    Note: Decimal-precision rates (e.g., 0.01) truncated to integer
    for storage; rates are recalculated from integer counts in Phase 4.

  STRATEGY C — Controlled Vocabulary Standardisation (Districts):
    Two district name variants corrected:
      MAKWANPUR   →   MAKAWANPUR
      CHITWAN     →   CHITAWAN
    Affected cells: 120 (across all district-month rows for these two
    districts over 60 months).

5. PHASE 3 — HL7 FHIR ARCHITECTURAL MAPPING AND STANDARDISATION
================================================================================

5.1 Conceptual Framework
--------------------------
Reports generated:
  - outputs/reports/06_FHIR_Measure_Standardisation_Report.txt
  - outputs/reports/11_DHIS2_FHIR_Mapping_Table.csv

The FHIR Clinical Reasoning Module was selected as the interoperability
framework for TB aggregate reporting. Within this module, two resource
types serve complementary roles:

  Measure (Definitional Resource):
    Acts as the computable quality standard. Defines the indicator name,
    scoring methodology, population groups (numerator, denominator,
    initial-population), and canonical URL.

  MeasureReport (Instance Resource):
    Holds the actual calculated results for a specific district over a
    specific reporting period. References the canonical Measure via URL,
    carries the count values, and expresses the computed score.

5.2 Scoring Type Architecture
-------------------------------
Two distinct FHIR scoring types were applied based on indicator semantics:

  RATIO Scoring (7 notification indicators):
    Applied when the numerator is NOT a strict mathematical subset of the
    denominator. Used for monthly TB case notification indicators where
    the numerator is a case count and the denominator is the total district
    population (CBS mid-year estimate, not the TB cohort).

    MeasureScore = numerator_count / denominator_count
    Rate per 100,000 = measureScore × 100,000

    Populations: numerator + denominator (2 population entries per group)

  COHORT Scoring (6 treatment outcome indicators):
    Applied for annual treatment cohort analysis. The initial-population
    is the registered cohort (pbc_reg) from which all outcome rates
    (cured, failed, died, ltfu, not_eval) are derived.

    Populations: initial-population (1 population entry per group)

6.3 Data Absent Reason (DAR) Implementation
---------------------------------------------
Under-reported months — BS 2078 Baishak, Jestha, and Asar — were identified
as months where DHIS2/IIHMS source data was not collected, yielding zeroes
in the raw CSV. These are methodologically distinct from true-zero reporting
and require explicit annotation per HL7 FHIR R4 standards.

Implementation:
  Extension URL  : http://hl7.org/fhir/StructureDefinition/data-absent-reason
  valueCode      : not-reported

  Applied to:
    - population[0]._count   (omits the count field; annotates with DAR)
    - group[0].measureScore._value (omits the score value; annotates with DAR)

  Affected indicators: All 7 ratio (notification) indicators for the 3 months
  Additional DAR     : relapse-total for BS 2079 Magh (1 empty cell in source CSV)
  Total DAR entries  : 22 across 780 MeasureReports

  Cohort indicators (pbc_reg, cured, failed, died, ltfu, not_eval) are NOT
  affected — these carry valid annual cohort values for all 60 months.

6.4 Bikram Sambat Calendar Extension
--------------------------------------
A custom FHIR extension was engineered to preserve Bikram Sambat calendar
context alongside mandatory Gregorian ISO 8601 period dates.

Extension canonical : https://iihms.gov.np/fhir/StructureDefinition/nepali-fiscal-period
Applied to          : MeasureReport.period (all 780 resources)

Sub-extensions:
  bs-year      : valueInteger  — BS year (e.g., 2078)
  bs-month     : valueString   — BS month name (e.g., "Shrawan")
  fiscal-year  : valueString   — Nepali fiscal year (e.g., "2078/79")

Gregorian date convention:
  period.start = {gregorian_year}-{gregorian_month}-16
  period.end   = {gregorian_year'}-{gregorian_month+1}-15

Nepal's fiscal year logic:
  Shrawan–Chaitra : FY {bs_year}/{bs_year+1 mod 100}
  Baishak–Asar    : FY {bs_year-1}/{bs_year mod 100}

This extension was formally defined in FHIR Shorthand (FSH) for the
forthcoming FHIR Implementation Guide (Phase 5, in progress).

8. LIMITATIONS AND KNOWN GAPS
================================================================================

  1. ART Coverage Gap (art_cov_pct):
     32.31% of records are missing ART coverage values. Classified as MNAR
     (Missing Not at Random). This variable was excluded from the FHIR
     transformation scope and documented as a formal structural gap.
     Implication: HIV/ART co-infection analysis cannot be performed with
     full statistical confidence.

  2. Under-Reported Months (BS 2078 Baishak, Jestha, Asar):
     Three months in the earliest year of coverage were not collected
     in the DHIS2 source. These months appear as zero values in the raw
     file. They have been handled through the FHIR Data Absent Reason
     mechanism (valueCode: not-reported) for all 7 notification indicators.
     Implication: FY 2078 notification analysis is incomplete for three months.

  3. Arithmetic Over-Reporting (12 rows):
     12 district-month records where the sum of outcome categories
     (cured + failed + died + ltfu + not_eval) exceeds the registered
     cohort (pbc_reg). These are identified as IIHMS-level data entry
     errors. They have been flagged but not modified, as correction
     requires authoritative source verification from NTP.

  4. Single-District FHIR Scope:
     FHIR resources cover Kathmandu District only. The harmonised dataset
     for all 13 Bagmati Province districts is available but not yet
     transformed into FHIR for all districts.

  5. FHIR Implementation Guide (Phase 5 — Incomplete):
     The formal FHIR Implementation Guide (IG) using FSH/SUSHI, including
     formal StructureDefinition profiles for Measure and MeasureReport,
     GitHub Actions CI/CD, and IG Publisher validation, is in progress
     and is not included in this report.

9. CONCLUSION
================================================================================

This project successfully delivers a reproducible, standards-compliant pipeline
that transforms five years of Nepal's IIHMS tuberculosis surveillance data into
HL7 FHIR R4 interoperable resources for Kathmandu District.

The four completed phases address the full data lifecycle from raw DHIS2 export
to queryable FHIR REST API: quality assessment with formal missingness mechanism
classification, a reproducible cleaning pipeline with Pandera schema enforcement,
a documented DHIS2-to-FHIR architectural mapping for 13 TB indicators, and
generation of 780 MeasureReport resources with 100% automated validation pass
rate.

The engineering decisions — ratio vs cohort scoring separation, Data Absent
Reason encoding for under-reported months, Bikram Sambat calendar preservation
through a custom FHIR extension, and the O(1) API lookup architecture — reflect
deliberate alignment with HL7 FHIR R4 specification requirements and Nepal's
public health reporting context.

The resulting system provides NTP and Bagmati Province health analysts with a
machine-readable, internationally exchangeable TB performance dataset while
preserving the local calendar context that is operationally essential for
Nepalese health information systems.
